# REVIEW-REQUEST.md — 第三者 AI / 人レビュアーへのレビュー依頼書

> このファイルは、Codex / Claude Code / Gemini CLI などの AI ツール、
> および人間のシニアエンジニア・エディタに対して、本リポジトリを
> **敵対的かつ批判的に** レビューしてもらうための依頼書です。レビュー
> の目的は「作者（TK2LAB）を責めること」ではなく、**このリポジトリを
> 読む人が安全に学び、時間を無駄にせず、誤った実装を真似しないための
> 第三者視点の検証** です。

---

## 0. リポジトリの目的と位置づけ（レビュー前に必読）

レビューを始める前に、次の 3 点を必ず踏まえてください。

1. **このリポジトリは「個人メモの共有」です。** 2026-04-18 時点で、
   TK2LAB と Codex が Windows 11 + WSL2 + Ubuntu + Bash + `codex-cli
   0.121.0` で画像生成を試したところ動いた、という覚書をそのまま
   公開しています。
2. **公式推奨の手順書ではありません。** 「こうすればこうなる」
   ではなく「このコマンドを私が走らせたら、こういう結果だった」と
   いうスタンスで書かれています。
3. **評価軸は「読者が安全に再現できるか」と「事実として正確か」**
   の 2 点です。「この書き方が最高」「このツールが推奨」といった
   ポジショニングに関する意見は評価対象外です。

したがって、レビューで「もっと完全なインストール手順を書くべき」
「runner をもっと高機能に」のような提案は対象外です。**読者にとって
の安全性・事実性・学習有益性を損なう可能性のある箇所** を洗い出す
ことに集中してください。

## 2. 評価軸（5 つ）

各ファイル・各主張・各コードブロックに対し、以下の 5 軸でチェックして
ください。発見は必ず **どの軸での指摘か** をラベル付けします。

| 軸 | 要点 |
| --- | --- |
| **R: 信頼性（Reliability）** | 書かれたコマンド・手順を第三者が真似したとき、再現性・安定性があるか。失敗条件は明示されているか。 |
| **Q: コード品質（Code Quality）** | `codex-image-batch.sh` と examples の JSON は、堅牢で読み取りやすいか。エラー処理・境界条件・命名は適切か。 |
| **S: セキュリティ（Security）** | 読者が実行しても、意図しない情報漏えい・意図しないファイル操作・権限昇格・外部送信などが起きないか。 |
| **D: ドキュメント信憑性（Doc Credibility）** | 本文中のクレームが事実と整合しているか。参照先 URL は存在し、内容が記述と合っているか。誇張・歪曲はないか。 |
| **E: 学習有益性（Educational Value）** | 初心者〜中級者が読んで、誤解なく学べるか。スキップしたほうが良い、危険な情報はないか。 |

---

## 3. レビュアー役割（6 つ）

各軸を評価するとき、少なくとも以下の 6 役割の視点から一度ずつ見て
ください。役割ごとに重視するポイントが違います。同じ問題でも役割に
よって評価が分かれることがあります（それ自体が有益な情報です）。

### 3.1 シニアエンジニア

- bash スクリプトの堅牢性、境界条件、失敗モード、タイムアウト、
  並行実行時の競合
- POSIX 準拠度、他環境への移植性
- エラーの粒度と `set -euo pipefail` の妥当性
- 「ここは書きすぎ」「ここは書かなすぎ」の判断

### 3.2 アーキテクト

- 役割分担（ランタイム / 設定 / I/O / ログ / サマリ）の境界が適切か
- 1 ファイル完結の是非、将来の拡張余地
- JSON schema の設計（`defaults` + `jobs` の merge 方式、相対パス解決
  の基準）
- fallback recovery（`~/.codex/generated_images`）の設計判断とリスク

### 3.3 デバッガー / テストエンジニア

- 具体的な失敗シナリオ（空 JSON、壊れた JSON、大量ジョブ、同時実行、
  中断、ディスク満杯、権限不足、UTF-8 以外、長すぎる prompt、ファイル
  名に特殊文字、ネットワーク切断）に対する挙動
- `--retry-count` / `--retry-delay` / `--stop-on-job-error` の効き方
- テスト容易性（静的チェックだけで確認できる範囲）

### 3.4 リサーチレポート作成者

- クレームと証拠の分離が明瞭か（例: 「私の環境では…」「公式では…」
  の区別）
- n=1 の範囲を越えた一般化が混入していないか
- 参照の選択バイアス（公式 URL 以外の一次情報を見落としていないか）
- 反証可能性と再現手順の明示度

### 3.5 入門 IT 解説書の執筆者・編集者・校正者

- 読者が途中でつまずきそうな用語・段落
- 初心者を置き去りにする飛躍や前提
- 「危険そうに見えるが説明不足」の箇所
- 用語の統一（runner / スクリプト、CLI / ツール、feature / 機能）
- トーンの一貫性（推奨風に読める言い回しが残っていないか）

### 3.6 カスタマーサポート品質管理者

- 「困ったとき」の動線が具体的で、独立して役立つか
- 原因不明の挙動を踏んだ読者のリカバリ経路
- 記述の「自分向け（作者向け）」から「読者向け」への切り替えの自然さ
- 他の読者を巻き込む表現（issue / PR 募集）が過度でないか

## 4. 重点的に疑ってほしいポイント（既知のリスク領域）

作者側で自己点検した時点で、次の箇所はレビュー時に特に注意してほしい
領域です。誇張・歪曲がないか、代替解釈がないかを追認してください。

| # | 観点 | 具体的に疑う点 |
| --- | --- | --- |
| 1 | 仕様の追認 | `codex exec --enable image_generation` と `--skip-git-repo-check` は、現行版でも正当なサブコマンド / フラグ構成か（`codex exec --help` / 公式 docs と突き合わせ） |
| 2 | 仕様の追認 | `~/.codex/config.toml` の `[features]` セクションによる有効化は、現行版でも動くか。再起動要否の記述がないが問題ないか |
| 3 | 仕様の追認 | `-i` を複数並べた場合の「1 枚目 base / 2 枚目 reference」という解釈は、CLI 実装仕様と整合するか（推測が混ざっていないか） |
| 4 | 画像サイズ | 3 バケットへの収束という主張が、OpenAI 公式の画像モデル docs と整合しているか。`gpt-image-1.5` の現時点での利用可否を URL で追認 |
| 5 | モデル名 | サンプル JSON で `codex_model: "gpt-5.4"` を指定しているが、現行 CLI が同名を受け付けるか。拒否される場合の症状が文書化されているか |
| 6 | fallback recovery | `find ... -newer "$marker_file"` による生成画像回収は、他の Codex プロセスや無関係な PNG を拾う競合リスクがないか。対策として必要十分か |
| 7 | タイムアウト | `codex exec` 実行に対して timeout が設定されていない点のリスク（ハング時に無限待ちする可能性） |
| 8 | 並行実行 | 同じ出力ディレクトリに対して同時に複数スクリプト実行が走った場合の競合（summary JSON、raw log、output PNG の衝突） |
| 9 | ログの機密性 | `raw_log_path` に書き出される Codex 出力に、ローカルパス・API エラー・認証情報の断片が含まれる可能性はないか |
| 10 | パス正規化 | `\\wsl.localhost\...` と `C:\...` の変換は十分か。エッジケース（UNC で `$` を含む、マルチバイト文字、`..` 混入）で失敗しないか |
| 11 | 設定ファイル | `~/.codex/config.toml` を手書きで書き換える記述で、既存 TOML を壊すリスクが説明されているか（重複セクション、構造） |
| 12 | Windows との境界 | PowerShell ネイティブ実行が対象外と繰り返しているが、誤って PowerShell で叩いた場合の「怪我しない」挙動になっているか |
| 13 | 参照 URL | 6 つの公式 URL が現時点で HTTP 200 を返し、本文記述と一致しているか。ドメイン変更、記事の削除、内容の大幅更新がないか |
| 14 | 誇張・断定 | 「〜です」「〜ます」の語尾に、観察レベルを越えた断定が混ざっていないか。「こう試した結果」の枠を越えていないか |
| 15 | 学習教材性 | 初心者が本文だけを読み進めた場合に、誤って危険なコマンド（`sudo`、`rm`、`--no-prompt`）を実行する動線が残っていないか |
| 16 | プロンプトの安全性 | サンプル JSON に含まれるプロンプトが、著名人・商標・特定キャラクターの生成を誘発しないか。OpenAI の利用ポリシーと整合するか |
| 17 | 依存コマンド | `jq` `python3` `realpath` `find` `sort` `awk` `grep` `cp` の依存が、Ubuntu LTS で標準 / apt 入手可か。ない環境での症状が明快か |
| 18 | 再現性 | 読者が記載のコマンドを貼り付けて実行したとき、失敗する確率の高いパターンはどこか |

## 5. 出力フォーマット（各レビュアーに遵守してほしい）

発見は次の形式で列挙してください。読み手（TK2LAB）が triage しやすい
ようにするのが目的です。自由形式の感想は後段の「総評」にのみ置いて
ください。

### 5.1 各指摘の書式

```
### Finding N — <短いタイトル>

- **Severity**: Critical / High / Medium / Low / Info
- **Axis**: R / Q / S / D / E（複数可）
- **Role**: Senior / Architect / Debugger / Researcher / Editor / CSQA（複数可）
- **File**: 相対パス
- **Locator**: 行番号または見出し（Markdown）
- **Evidence**: 実コマンド・該当引用・公式 URL・実行結果など
- **Impact**: 読者にどう影響するか（時間浪費 / 誤解 / 実害 / セキュリティ）
- **Suggested fix**: 具体的な差分案（diff または書き換え文）
- **Confidence**: High / Medium / Low
```

### 5.2 Severity の目安

- **Critical**: そのまま読むと読者が実害を被る、または重大な誤解を
  与える（例: 秘密情報を漏らす手順、破壊的コマンドの推奨）
- **High**: 再現性が著しく損なわれる、根拠不足の断定、公式と矛盾する
  記述、未文書化の挙動依存
- **Medium**: 読みにくさ、不整合、トーンずれ、用語ぶれ、学習動線の
  ねじれ
- **Low**: 言い回し・体裁・typo・細かな表記ゆれ
- **Info**: 参考情報、代替案、補強すべき周辺情報

### 5.3 全体レビューの総評

個別指摘の後に、次の要素を 1 画面に収まる分量で書いてください。

```
## 総評

- 対象リポジトリが、その目的（2026-04-18 時点の個人検証の共有）に
  照らして十分に機能しているか: Yes / No / Partial
- 読者が時間を無駄にするリスク: Low / Medium / High
- 読者が誤った実装を真似するリスク: Low / Medium / High
- セキュリティ・プライバシーに関わる残留リスク: Low / Medium / High
- もっとも優先して修正すべき 3 点（重要度順）
- 現状の優れている点 3 点（過剰に持ち上げず、事実に基づいて）
- このリポジトリが「読む価値・学ぶ価値・活用する価値があるか」
  についての総合判定と 2〜3 行の理由
```

## 7. TK2LAB 側の反映ワークフロー

1. 各 AI / レビュアーから Finding 一覧と総評を受領
2. Severity が Critical / High のものを最優先で triage
3. 同じ内容の指摘を AI 横断で集計（3 レビュアーのうち 2 以上が
   指摘した項目は優先度を 1 段上げる）
4. 確定した修正は `git commit` で反映。コミットメッセージに
   「per adversarial review (Codex/Claude Code/Gemini)」等を
   明示しても可
5. 反映できないもの・保留中のものは `MAINTAINER-NOTES.md` に
   理由付きで転記
6. レビュー結果そのもの（生の Finding 一覧）はリポジトリに含めない。
   必要なら別リポ / ローカルで保管

## 8. このレビューの「やらない」リスト

次の依頼は本書の対象外です。

- スクリプトをより高機能にする（並列実行、DAG、CI 連携等の追加提案）
- OS / 配布形態を変える提案（Docker 化、Brew、バイナリ配布等）
- デザイン・装飾の改善提案
- 完全な入門チュートリアルへの書き換え
- 作者の考え方・ブランドへの論評

あくまで **読者の安全 / 事実性 / 学習有益性** の観点で、足りないと
判断される情報を列挙することに集中してください。
