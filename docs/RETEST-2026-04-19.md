# 2026-04-19 Re-test Summary

In this environment, the 2026-04-19 re-test confirmed that both image
generation and image editing still worked with `codex-cli 0.121.0` in WSL2
Ubuntu on Windows 11.

However, the generated PNGs were stored under
`~/.codex/generated_images/<session-id>/` rather than the
working-directory location expected by the prompt or helper script.

For that reason, the helper flow in this repository now includes a recovery
step that copies the generated files from the matching session directory after
each Codex CLI run.

This re-test summary records only what was confirmed in this environment.
Behavior may differ with other Codex CLI versions, auth state, runtime setups,
or future product changes.

The full evidence bundle remains local/private because it contains raw logs,
machine-specific paths, and other local-only artifacts. The public re-test
record intentionally excludes raw logs, local-machine-specific paths, private
assets, and credentials.

## What was checked

1. Whether image generation worked through `codex exec` when
   `image_generation=true` was already persisted in `~/.codex/config.toml`
2. Whether `--enable image_generation` changed the behavior
3. Whether a one-image edit worked through `-i`
4. Whether `--full-auto -c sandbox_workspace_write.network_access=true`
   changed the outcome

## What held up in this environment

- Generation worked.
- Editing worked.
- `--enable image_generation` was accepted by the CLI.
- `--enable image_generation` behaved like a practical no-op once config
  already had `image_generation=true`.
- The extra `network_access=true` setting did not change the outcome in this
  run.

## What changed relative to the original write-up

- The generated PNGs were not written to the working-directory paths expected by the prompt or helper flow.
- The actual files were found under
  `~/.codex/generated_images/<session-id>/ig_*.png`.
- The model sometimes printed an `Output path: ...` line that did not exist on
  disk, so the printed path must be treated as untrusted unless the file is
  actually present.

## Public-safe workflow to reproduce

```bash
printf '%s\n' 'Use the built-in image generation capability only. Generate a square 1:1 image of a blue sphere on a white background. No text, no logo, no watermark.' \
| codex exec --skip-git-repo-check -
```

Then read the `session id` from the run log and copy the file out manually:

```bash
session_id="<session-id-from-log>"
cp ~/.codex/generated_images/"$session_id"/*.png ./recovered-output.png
```

## Repository changes made because of this re-test

- The READMEs and QUICKSTART docs now describe internal-storage recovery
  explicitly.
- The helper script now records `session_id` in summaries and prefers
  session-specific recovery before falling back to a broader filesystem scan.
- Sample JSON and input notes now include a multi-reference character example.
- CI validates the helper script and example JSON on every push and pull
  request.

## How to read this note

The goal of this document is not to define official Codex CLI behavior. It
records what was re-tested successfully in one environment, plus the
operational detail that was easy to miss: image recovery from
Codex-managed storage was still necessary.

If you try the same flow elsewhere, verify the Codex CLI version, official
documentation, auth state, and the actual location of generated files on your
own machine.
